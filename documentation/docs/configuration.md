# Configuration

Pipeline behaviour is controlled by two configuration sources:

1. **`.pipeline/config.yml`** in your repository â€” non-secret step parameters
2. **GitHub repository secrets** with a `PIPER_` prefix â€” credentials and other sensitive values

The `piper` CLI merges these at runtime: a `PIPER_<paramName>` environment variable always takes precedence over the same parameter in `config.yml`.

## `.pipeline/config.yml`

Place this file in the root of your repository. Use it for non-secret configuration that can be committed to source control.

```yaml
general:
  abapAddonAssemblyKitEndpoint: "https://aakaas.example.com"

steps:
  abapEnvironmentCloneGitRepo:
    repositories: repositories.yml

  abapEnvironmentRunATCCheck:
    atcConfig: atcConfig.yml

  abapEnvironmentRunAUnitTest:
    aUnitConfig: aUnitConfig.yml
```

### Configuration hierarchy

Parameters are resolved in the following priority order (highest first):

1. `PIPER_<paramName>` environment variable (GitHub secret)
2. Step-level entry under `steps:` in `config.yml`
3. `general:` section in `config.yml`
4. Built-in defaults from `resources/default_pipeline_environment.yml`

## GitHub secrets

Credentials are passed as environment variables with the `PIPER_` prefix. Every `piper <step>` command reads them automatically â€” no explicit flag passing required.

| Secret name | Parameter | Used by |
|-------------|-----------|---------|
| `PIPER_ABAP_ADDON_ASSEMBLY_KIT_COOKIE` | `abapAddonAssemblyKitCookie` | All `abapAddonAssemblyKit*` steps |
| `PIPER_USER` | `user` | All `abapEnvironment*` steps, BTP steps |
| `PIPER_PASSWORD` | `password` | All `abapEnvironment*` steps, BTP steps |
| `PIPER_CF_USER` | `cfUser` | CF provisioning steps |
| `PIPER_CF_PASSWORD` | `cfPassword` | CF provisioning steps |
| `PIPER_BTP_API_CREDENTIALS_ID` | `btpApiCredentialsId` | BTP provisioning steps |

Add secrets at **Settings â†’ Secrets and variables â†’ Actions â†’ New repository secret**.

## Reusable workflow inputs

Stage toggles and non-secret config are passed as `with:` inputs in the consumer workflow. The scaffold generated by `piper initGithubActions` pre-fills sensible defaults:

```yaml
jobs:
  pipeline:
    uses: SAP/jenkins-library/.github/workflows/abapEnvironmentPipeline.yml@main
    with:
      addonDescriptorFileName: addon.yml
      repositories: repositories.yml
      stage_atc: true
      stage_aunit: true
      stage_integrationTest: false
      stage_confirm: true
      confirmEnvironment: production
    secrets:
      PIPER_abapAddonAssemblyKitCookie: ${{ secrets.PIPER_ABAP_ADDON_ASSEMBLY_KIT_COOKIE }}
      PIPER_user: ${{ secrets.PIPER_USER }}
      PIPER_password: ${{ secrets.PIPER_PASSWORD }}
```

See the full list of available inputs in the [reusable workflow source](https://github.com/SAP/jenkins-library/blob/main/.github/workflows/abapEnvironmentPipeline.yml).

## ATC configuration

Create `atcConfig.yml` to specify which objects ATC should check:

```yaml
objectSet:
  softwarecomponent:
    - name: /DMO/MY_COMPONENT
```

Pass the filename via `.pipeline/config.yml`:

```yaml
steps:
  abapEnvironmentRunATCCheck:
    atcConfig: atcConfig.yml
```

## AUnit configuration

Create `aUnitConfig.yml` to specify which objects AUnit should test:

```yaml
objectSet:
  softwarecomponent:
    - name: /DMO/MY_COMPONENT
```

Pass the filename via `.pipeline/config.yml`:

```yaml
steps:
  abapEnvironmentRunAUnitTest:
    aUnitConfig: aUnitConfig.yml
```

## Common pipeline environment

The `piper` CLI persists state between steps in the `.pipeline/` directory (the *common pipeline environment*, CPE). In GitHub Actions each job runs on a fresh runner, so the reusable workflow uploads `.pipeline/` as an artifact after every job and downloads it at the start of the next. This is handled automatically â€” no action needed on your part.
